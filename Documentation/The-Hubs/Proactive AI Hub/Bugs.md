# Proactive AI Hub: Bug Report & Known Issues

## 1. Overview
This document tracks identified bugs, logic flaws, and edge cases within the Proactive AI Hub codebase. These issues range from critical execution flaws to minor UI inconsistencies and must be addressed to ensure system stability.

## 2. High Priority / Critical Bugs

### 2.1 Scheduler Action Hardcoding
- **Location:** `app/Console/Commands/ProactiveSchedulerCommand.php` (Lines 48-58)
- **Description:** The command currently hardcodes the execution logic for the `notify` action directly inside the loop. If a trigger contains any action *other* than `notify` (e.g., `reply`, which is generated by the NLP parser in `NlpParserService`), the scheduler will silently ignore it but still mark the trigger as `completed`.
- **Impact:** Critical. Users creating rules like "reply with..." will find that their rules are parsed and marked as complete, but no actual reply is ever sent.
- **Proposed Fix:** Implement an Action Factory or a switch/match statement that handles different action keys in the `actions` payload, delegating execution to specific handler classes.

### 2.2 Time Parsing Logic Flaws
- **Location:** `app/Services/Proactive/NlpParserService.php` (`parseTime` method)
- **Description:** The time parser contains logical flaws in handling AM/PM. Specifically, `if ($ampm === 'am' && $hour === 12) { $hour = 0; }` handles midnight correctly, but `if ($ampm === 'pm' && $hour < 12) { $hour += 12; }` handles afternoon. However, the regex `/at (\d+)\s*(am|pm)?/i` makes AM/PM optional. If a user types "at 15", the parser will default to `am` (because `isset($matches[2]) ? strtolower($matches[2]) : 'am'`), resulting in 15 AM, which Carbon might misinterpret or throw an error on when `setTime(15, 0)` is called.
- **Impact:** High. Time-based rules may execute at entirely incorrect times.
- **Proposed Fix:** Use PHP's built-in `strtotime()` combined with Carbon for more robust parsing, or rely on a dedicated NLP time-parsing library.

## 3. Medium Priority Bugs

### 3.1 Orphaned Event Triggers
- **Location:** `app/Services/Proactive/NlpParserService.php` vs `ProactiveAIController.php`
- **Description:** When an `event_based` rule is parsed, it returns an array without a `next_run_at` value. The controller `storeRule` method only creates a `ProactiveTrigger` if `$parsed['type'] === 'time_based'`. Therefore, event-based rules are stored in `EcaRule`, but there is currently no system mechanism (event listener) in place to actually generate triggers for them when the event occurs.
- **Impact:** Medium. Event-based rules currently do nothing; they are essentially dead data.
- **Proposed Fix:** Implement Laravel Event Listeners that monitor system events, check active `EcaRule` records, and dynamically generate `ProactiveTrigger` records at runtime.

### 3.2 Transaction Integrity in Scheduler
- **Location:** `app/Console/Commands/ProactiveSchedulerCommand.php`
- **Description:** The scheduler command loops over triggers and updates statuses without utilizing database transactions. If the server crashes mid-loop, some triggers might be executed but not marked as completed, leading to duplicate executions on the next cron run.
- **Impact:** Medium. Potential for duplicate notifications or actions.
- **Proposed Fix:** Wrap the trigger processing logic inside a `DB::transaction()` block.

## 4. Low Priority / UI Bugs

### 4.1 Mocked UI Creation
- **Location:** `resources/views/hubs/proactive-ai.blade.php` (Line 321, `createRule` function)
- **Description:** The "New Rule" modal's submit button calls a JavaScript function `createRule()` which merely simulates a loading delay and shows an alert. It does not actually perform an AJAX POST request to `ProactiveAIController@storeRule`.
- **Impact:** Low (Development state). The feature is functionally disconnected from the backend in the UI.
- **Proposed Fix:** Replace the `setTimeout` mock in `createRule()` with an actual `fetch()` or `axios` call to the API endpoint, passing the textarea value.

### 4.2 Missing Validation on Rule Update
- **Location:** `app/Http/Controllers/ProactiveAIController.php`
- **Description:** The `toggleRule` method allows toggling the `is_active` state but does not validate if the rule exists properly or if the user owns the rule (missing Authorization).
- **Impact:** Low to Medium security risk depending on system architecture.
- **Proposed Fix:** Add route model binding, authorization policies, or ownership checks before allowing a toggle.
